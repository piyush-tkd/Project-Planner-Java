1. Executive Summary

After a live side-by-side comparison of Wrike's PMO template against the Engineering Portfolio Planner (EPP), the clear picture is this: EPP has more raw data depth than Wrike in every engineering-specific dimension — capacity math, POD planning, sprint/release tracking, and Jira integration — but Wrike wins decisively on usability, configurability, workflow design, and multi-user roles.

Core strategic finding EPP is not a Wrike clone — it should not try to be. EPP's moat is engineering-specific intelligence (capacity math, POD matrices, DORA, Jira actuals, worklog, hiring forecast) that Wrike simply doesn't have. The opportunity is to match Wrike's usability patterns while keeping EPP's depth advantage, and architect for multi-org SaaS from the start.
Where EPP is ahead of Wrike POD-level capacity planning, DORA metrics, Jira actuals vs estimated, sprint/release calendar layers, workload heatmap by POD/month, hiring forecast, scenario simulators, NLP query interface, support queue monitoring, resource bookings, and budget/CapEx tracking.
Where Wrike is ahead of EPP Role-based dashboards (Executive / Manager / IC), inline task management and subtasks, configurable columns per view, request intake forms, timesheets & approval workflows, ideas Kanban (innovation funnel), risk & issue registry with scoring, whiteboard collaboration, RACI charts, and project scoring rubrics.

3. Design Gaps & Improvements

3.1 Navigation — Too Many Top-Level Pages

EPP currently has 40+ distinct sidebar items across 8 collapsible sections. Wrike's sidebar has 7 global items (Inbox, Request forms, Workload, Timesheets, Whiteboards, More + Spaces). The difference is massive cognitive load. Users don't know where to start.

Problem: The sidebar is a feature list, not a workflow. First-time users are lost. Power users get fatigued. The navigation must reflect jobs-to-be-done, not the database schema.
Solution: Restructure navigation into 5 top-level workflow zones — Monitor, Plan, Execute, Report, Admin — and consolidate sub-pages into tabbed views within each zone. Target: max 12 sidebar items visible at any time.

3.2 Table Column Configurability

Wrike lets users toggle any field on/off, reorder columns, and save named views (Projects | Financials | RACI | Scoring). EPP has fixed columns. For a SaaS product serving multiple orgs, every org will want different columns. This is a blocker.

Action needed: Implement a column configurator (show/hide toggle panel) for all data tables. Save preferences per user per page. Eventually, make custom fields per org configurable in the Settings layer.

3.3 Empty States & Onboarding

Wrike's empty states guide you: "Create a workload chart", "Balance your team's workload at a glance", tutorial videos inline. EPP shows blank tables or "No data". For a new org onboarding, this is a critical gap.

3.4 Dashboard — Header Title Changes per Board

When switching between Team Dashboard / Executive Board / Analyst Board in EPP, the page title updates correctly ("Team dashboard", "Executive board") — this is good Wrike parity. Continue this pattern on other multi-view pages.

3.5 Status Color System

EPP's STATUS_META palette is solid. Wrike uses more muted, pastel-adjacent colors. Our colors are slightly more saturated — consider reducing saturation 10–15% for a more enterprise feel, especially on the status pills in table views.

3.6 Project Detail View / Slide-Over Panel

In Wrike, clicking a project opens a full slide-over panel with: description, custom fields, subtasks, comments, files, activity log. In EPP, clicking a project navigates to a detail page. The pattern works but the detail page currently lacks: comments, activity log, file attachments, and inline field editing.

3.7 Typography & Density

EPP's dark navy header + aqua accent is distinctive but not Wrike-style (Wrike uses near-white sidebar). Consider offering a light sidebar mode as a user preference — important for accessibility and enterprise buyer perception.

3.8 Mobile / Responsive Gaps

Wrike has a full mobile app. EPP has no responsive breakpoints documented. For SaaS, mobile-friendly at least for dashboards and approval workflows is expected by buyers.

5. SaaS & Multi-Org Readiness

The vision is clear: today Baylor Genetics, tomorrow any engineering org. Here is a structured assessment of what it takes to make EPP a true multi-tenant SaaS product.

🏢 Multi-Tenancy (Organizations / Workspaces) Missing

  • Every entity in the DB (projects, resources, PODs, sprints, releases) must be scoped to an org_id
  • Authentication must support per-org user pools (currently single-org users table)
  • Backend APIs must filter all queries by org context from JWT token claims
  • Flyway migrations need to support multi-tenant schema strategy (shared schema + org_id OR schema-per-org)
  • Org onboarding flow: create org → invite admin → configure reference data → seed with sample data

🔐 Role-Based Access Control (RBAC) Partial

  • Currently: admin / user roles at the app level only
  • Needed: Org Admin (full settings, user management), Portfolio Manager (all projects), Team Lead (own POD only), IC / Viewer (read-only dashboard)
  • Page-level permissions already exist in the DB (page_permissions table) — extend to role-based visibility
  • Wrike's IC dashboard shows only "My tasks / My projects" — EPP needs user-scoped data views

⚙️ Org-Level Configuration Missing

  • Custom project statuses per org (not hard-coded ACTIVE / IN_DISCOVERY / etc.)
  • Custom priority levels (P0–P3 vs High/Medium/Low vs custom)
  • Custom fields on projects, resources, PODs
  • Custom POD types / team structures
  • Org branding: logo, accent color, org name (currently hard-coded "Baylor Genetics")
  • Configurable fiscal year start month (affects all capacity/demand calculations)
  • Toggle modules on/off: Timesheets, Jira Integration, NLP, Support Queue, etc.

💳 Billing & Plan Management Not started

  • Plan tiers: Starter (1 org, 10 users), Team (50 users, all modules), Enterprise (unlimited, SSO, audit log, SLA)
  • Feature flags per plan: NLP/AI, Simulators, DORA, Jira Integration are natural premium features
  • Metering: users, projects, API calls
  • Stripe integration for self-serve billing; enterprise = contract

🔗 Integrations & API Partial

  • Jira integration exists but is org-specific (credentials in settings) — needs per-org credential vault
  • Webhook support for outbound events (project status changes, capacity alerts)
  • Public REST API for orgs to push/pull data (import/export beyond CSV)
  • SSO / SAML for enterprise orgs (currently username/password only)
  • OAuth2 for Jira, GitHub, Slack integrations

🌐 Deployment & Infrastructure Not production

  • Currently: localhost dev server + local PostgreSQL. No production deployment.
  • Needed: Dockerize Spring Boot + Vite frontend → deploy to cloud (Render, Railway, Fly.io, or AWS ECS)
  • Managed PostgreSQL (RDS / Supabase / Neon) with connection pooling
  • Environment-based config (application-prod.yml vs dev)
  • HTTPS, custom domain, CDN for frontend assets
  • Automated backups, audit log persistence, GDPR data deletion

🎨 White-Label / Org Branding Missing

  • "Baylor Genetics" is hard-coded in the login screen and header
  • Needs: org name, logo, and primary accent color stored per org and applied at runtime via CSS variables
  • This single change makes the product visually multi-org ready immediately

6. Critical Workflow Gaps

6.1 Project Intake Workflow

Wrike's Request Forms create a structured intake funnel: stakeholder submits form → lands in Requested folder → PM reviews → promotes to Active → assigns resources. EPP has no intake workflow — projects are only created manually by admins. This is a critical missing workflow for any org using EPP.

Proposed flow: New Request Form → Ideas Board (Kanban, unscored) → Portfolio Manager scores & approves → converts to Project record → assigns POD, estimates, releases.

6.2 Approval & Status Change Workflow

Currently any user can change any project status with no approval gate. For enterprise orgs, moving a project from In Discovery → Active or Active → On Hold should trigger a notification and optionally require a second approval.

6.3 Resource Booking Conflict Detection

EPP has a Resource Bookings page but no conflict detection (i.e., if the same person is booked at 200% in a sprint, there's no visual alert inline in the project planner). Wrike's Workload view shows 150% in red immediately.

6.4 Sprint Planning → Release → Deployment Pipeline

EPP has Sprint Planner, Sprint Calendar, Release Calendar, Release Versions — but the workflow connection between them is unclear. Does completing a sprint auto-update the release? Is there a pipeline view? This end-to-end workflow should be documented and surfaced in the UI.

Everything else is a feature sprint on top of that foundation.
